# Prompt Operations

## 1) 목적
- 프롬프트를 코드처럼 버전관리하고, 변경 영향을 통제한다.

## 2) 저장/버전 정책
- 프롬프트는 Git으로 버전관리한다.
- 저장 위치: `prompts/` 디렉토리 (레포 루트)
- 파일 포맷: Markdown (`.md`)
- 명명 규칙: `<용도>-<스코프>.md` (예: `feature-prescription.md`, `review-security.md`)
- 파일 상단에 frontmatter로 메타 정보를 둔다:

```markdown
---
purpose: 무엇을 만드는 프롬프트인가
scope: user|pet|prescription|... (PR scope 화이트리스트와 동일)
owner: 담당자 GitHub 핸들
last_reviewed: YYYY-MM-DD
---
```

- 변경은 PR로 제출하고 최소 1명 승인 후 반영한다.
- 프롬프트 변경 사유/기대효과/리스크를 PR 본문에 기록한다.
- PR 제목은 `type(scope): 제목` 형식을 사용한다 (`type`은 보통 `chore` 또는 `docs`).

## 3) 프롬프트 템플릿 (기본)
- 목표: 무엇을 만들거나 수정할지
- 제약: 아키텍처/보안/성능/스타일
- 입력: 참고 파일/스키마/요구사항
- 출력: 코드, 테스트, 변경 요약, 검증 방법
- 금지: 시크릿 출력, 민감정보 노출, 근거 없는 가정
- 필수: 가정값 사용 시 `가정/영향/검증방법` 3가지를 반드시 명시

## 4) 검증 루프
1. 소규모 변경으로 시범 실행
2. 결과 품질/정확성 확인
3. 실패 패턴 기록
4. 프롬프트 개선 및 재실행

## 5) 롤백 기준
- 품질 지표 악화(빌드 실패율 증가, 회귀 증가)
- 보안 정책 위반 가능성 발견
- 도메인 규칙(DDD/OOP) 위반 증가

## 6) 운영 체크리스트
- 변경 전/후 프롬프트 diff 확인
- 재현 가능한 입력 샘플 보관
- 실패 사례를 다음 프롬프트 개선에 반영
- 체크리스트 미완료 PR은 머지하지 않는다.
